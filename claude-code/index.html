Claude Code 入门

先让它理解项目，再让它碰代码。你需要的不是更多功能，而是一套不会失控的协作顺序。

本页目标：完成一次“读规则 → 锁范围 → 小步修改 → 真实验证”的最小闭环。

01 · Claude Code 是什么

Claude Code 是运行在开发环境里的 AI 编程 Agent。它不只是回答代码问题，还能在获得允许后读取项目文件、修改代码、运行命令并检查结果。

真正重要的差别不是“它能写多少代码”，而是它能否接触真实上下文，并在同一个工作区里把任务推进到可验证状态。

一句话定义Claude Code 是能进入真实项目、按约束执行任务的 AI 编程协作者。

02 · 它和聊天工具有什么不同

工具形态常见输入常见输出你的主要工作
网页聊天复制的代码与报错建议或代码片段手工定位、粘贴和验证
代码补全当前文件与光标上下文局部补全决定结构并组合代码
Claude Code项目、规则和任务文件改动与验证结果定义目标、授权并验收

它更像坐在项目旁边的执行者，但最终拍板、业务判断和结果验收仍然属于你。

03 · 启动与环境确认

先在终端里检查命令是否可用，再进入你的项目目录启动。

# 检查安装
claude --version

# 进入真实项目
cd 你的项目目录
claude

如果版本命令不存在，先完成官方安装；不要在不确定来源的脚本里输入账号、密钥或支付信息。

启动后先别改第一轮只让它读取规则和结构。先检查它理解的项目目标是否准确。

04 · 第一条有效指令

先读取项目规则和 README，只做只读探索。
请告诉我：
1. 项目目标是什么
2. 与当前任务相关的文件有哪些
3. 哪些现有改动必须保留
4. 最小可交付改动是什么
5. 修改后准备运行什么验证

这段话不是万能提示词。它的工作是强迫任务先拥有上下文、范围和验收，再进入实现。

05 · 探索、计划与修改

探索

让 Claude Code 找到规则文件、入口文件、相关测试和当前改动。没有找到关键上下文时，不要急着开工。

计划

涉及多个文件、外部依赖、数据迁移或发布时，要求它先列步骤。单文件且边界明确的修复可以直接做。

小步修改

一次只处理一个清晰结果。修改后马上运行最接近风险点的验证，不要攒到最后一次性排错。

请只修改 app/ 和 tests/ 中与当前任务相关的文件。
保留工作区里已有的其他改动。
完成后运行对应测试，并报告未验证项。

06 · 上下文与项目规则

项目规则文件的作用，是把每次都要解释的约束固定下来，例如目录结构、测试命令、禁止动作和交付格式。

  • 规则只写长期有效的约束，不记录一次性聊天；
  • README 面向使用者，规则文件面向执行任务的 Agent；
  • 具体任务仍然要说明目标和完成标准；
  • 规则冲突时，优先采用距离当前文件更近、更加具体的要求。

07 · 权限与安全

  • API Key 放在环境变量，不写入代码、截图或公开仓库；
  • 安装依赖、联网、删除文件、推送和发布前确认影响；
  • 已有工作区改动默认属于用户，不擅自重置；
  • 遇到高风险动作时，宁可缩小范围，也不要用更宽权限绕过去；
  • AI 生成代码仍然要经过测试、预览或人工检查。

08 · 完成标准

一次任务至少要留下以下证据：

  • 改了什么，以及为什么改；
  • 实际运行了哪些验证命令；
  • 哪些测试通过或失败；
  • 哪些外部状态没有验证；
  • 下一位接手者如何复现。

没有运行证据，就不要写“已完成”；没有真实交付，就不要继续刷教程。
