Codex 实战入门

同一个 Codex 可以出现在不同入口里。不要先纠结界面，先学会把目标、规则和验证交给它。

本页目标：让 Codex 完成一次有边界、有测试、有交付报告的小任务。

01 · Codex 与四种入口

Codex 是 OpenAI 的 AI 编程 Agent。常见入口包括桌面应用、命令行、IDE 集成和云端任务。它们共享相似的执行逻辑，但适合的工作场景不同。

入口适合场景主要特点
桌面应用多任务与可视化审阅适合集中查看进度和改动
命令行真实项目与自动化最接近文件、测试和脚本
IDE 集成边写边改编辑器上下文切换少
云端任务异步执行任务可以离开本机继续运行

02 · 该从哪里开始

如果你第一次使用，选择最容易看到项目文件和验证结果的入口即可。目标不是一次把四种入口学完，而是跑通同一套动作。

推荐判断想做真实项目，优先命令行或桌面应用；只想理解工作流，先从当前最顺手的入口开始。

03 · 启动第一个任务

进入一个体量小、可以安全测试的项目。先让 Codex 读取规则，不要第一句话就要求重构全部代码。

读取 AGENTS.md、README 和相关测试。
目标：完成一个可运行的最小闭环。
要求：保留无关改动；修改后运行测试；
不要把文档存在当成运行通过。

如果任务涉及多个文件或外部依赖，让它先给出计划；如果只是一个明确的小修复，可以直接执行并立即验证。

04 · 用 AGENTS.md 固定项目规则

AGENTS.md 是写给编码 Agent 的项目说明。它可以记录项目结构、测试命令、文件边界和协作约定。

# 项目说明
- 目标：维护一个纯静态教程站
- 运行：直接打开 index.html
- 验收：python test_site.py
- 禁止：不要删除用户已有资料
- 发布：本地验收后再同步远程

规则文件不能代替当前任务。每次仍要告诉 Codex：这次要交付什么，以及什么时候算完成。

05 · 探索、计划与实现

探索不是拖延

先找到真正相关的文件、现有测试和未提交改动。探索的终点，是能准确说出修改范围。

计划不是仪式

只有当计划能说明依赖顺序、验证方式和风险时，它才有价值。单文件改字不需要写十步计划。

实现要小步闭环

优先完成最小可运行结果，马上测试。复杂任务也要拆成一系列能独立验证的小结果。

06 · 把证据放进交付

最终报告不能只说“已经优化”。它应该包含可以复核的事实：

  • 修改文件清单；
  • 实际执行的命令；
  • 测试通过数量；
  • 未验证的浏览器、账号或线上状态；
  • 下一步需要谁做什么。
错误示范“代码已完成，应该可以运行。”
有效交付“运行了 12 项链接检查，全部通过；线上缓存尚未验证。”

07 · 沙箱与审批

权限越宽，犯错半径越大。能在当前目录完成的任务，不要为了省一步就扩大到整台电脑。

  • 读文件和本地测试通常可以直接做；
  • 联网安装、修改系统目录和外部写入需要明确确认；
  • 删除、覆盖和发布前要确认目标路径与回滚点；
  • 遇到权限失败时，先找更小范围的替代路径，而不是绕过审批。

08 · 完成标准

  • Codex 能准确复述任务目标与约束；
  • 修改只落在约定文件中；
  • 结果能够构建、运行或被自动检查；
  • 测试来自实际执行，而不是推测；
  • 未完成和未验证内容被明确标记。

工具只是杠杆。你真正要建立的是一套“下一位接手者也能复现”的生产流程。
